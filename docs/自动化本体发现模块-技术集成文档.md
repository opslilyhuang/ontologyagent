## 1. 模块定位与价值

### 1.1 现有系统概述

现有系统包含以下核心模块：
- **人工数据导入**: 手动上传和配置数据源
- **数据编排**: 人工定义本体结构（类、属性、关系）
- **本体管理器**: 管理和维护本体定义
- **智能问答**: 基于本体的知识问答系统

### 1.2 新增模块定位

**自动化本体发现模块**是一个**独立的一级菜单模块**，提供从数据源到本体的自动化流程：

```
数据接入 → 自动分析 → 本体发现 → 人工审核确认 → 数据清洗配置 → 导入到本体管理器
```

**核心价值**：
- 🚀 **效率提升**: 自动识别数据结构，减少90%的人工编排工作
- 🎯 **降低门槛**: 无需深度本体知识，系统自动推荐类和关系
- 🔄 **快速迭代**: 支持批量数据源分析，快速构建知识图谱
- 🧹 **数据质量**: 内置数据清洗引擎，确保导入质量

### 1.3 与现有系统的关系

### 3.2 自动化本体分析与发现

#### 3.2.1 分析流程

**触发方式**:
- 单个数据源分析: `POST /api/v1/data-sources/{id}/analyze`
- 批量数据源分析: `POST /api/v1/data-sources/batch-analyze`

**分析阶段**:

```
阶段 1: 数据理解 (5秒)
├─ 加载数据源
├─ 统计行列数
└─ 基础数据概览

阶段 2: 实体识别 (10秒)
├─ 列类型推断 (类型推断引擎)
├─ 主键识别
├─ 字段语义理解
└─ 生成实体类 (Class)

阶段 3: 关系发现 (10秒)
├─ 外键检测
├─ 跨表关系推断
├─ 关系置信度计算
└─ 生成关系定义 (Relation)

阶段 4: 本体生成 (5秒)
├─ 本体结构优化
├─ 中文标签生成
└─ 完成本体定义
```

**总耗时**: 约 30 秒/数据源

#### 3.2.2 类型推断引擎

**位置**: `backend/app/engines/type_inference.py`

**推断逻辑**:

```python
# 伪代码示例
def infer_type(column_data):
    if is_numeric(column_data):
        if all_integers(column_data):
            return "int32"
        else:
            return "float64"
    elif is_datetime(column_data):
        return "datetime"
    elif is_email(column_data):
        return "email"
    elif is_url(column_data):
        return "url"
    elif is_enum(column_data):  # 唯一值少于10个
        return "enum"
    else:
        return "varchar"
```

**支持的数据类型**:
- `int32`, `int64`, `float64`, `boolean`
- `varchar`, `text`, `enum`
- `date`, `datetime`, `timestamp`
- `email`, `url`, `uuid`, `ip`, `json`

#### 3.2.3 关系发现引擎

**位置**: `backend/app/engines/relation_inference.py`

**推断策略**:

1. **外键检测**
   - 字段名匹配 (如 `user_id` → `User.id`)
   - 值域检测 (目标字段值是否存在于源字段)
   - 置信度计算

2. **语义关系**
   - 通过大模型推断字段间的业务关系
   - 例如: `订单.客户名称` → `客户.姓名`

3. **关系类型**
   - `1:1` (一对一)
   - `1:n` (一对多)
   - `n:1` (多对一)
   - `n:m` (多对多)

**输出示例**:
```json
{
  "source_class": "Order",
  "target_class": "Customer",
  "relation_name": "belongs_to",
  "cardinality": "n:1",
  "source_field": "customer_id",
  "target_field": "id",
  "confidence": 0.95
}
```

```python
class DataCleaningEngine:
    def clean(self, df: pd.DataFrame, config: dict):
        """
        执行数据清洗
        返回: (清洗后的DataFrame, 失败记录列表)
        """
        cleaned_df = df.copy()
        failed_records = []

        for operator in config['operators']:
            cleaned_df, failures = self._apply_operator(
                cleaned_df,
                operator
            )
            failed_records.extend(failures)

        return cleaned_df, failed_records
```

#### 导入服务

**位置**: `backend/app/services/import_service.py`

**核心方法**:
```python
class OntologyImportService:
    async def import_ontology_data(
        self,
        db: AsyncSession,
        ontology_id: str,
        cleaning_config: dict
    ) -> str:
        """
        导入本体数据
        返回: import_log_id
        """
        # 1. 创建 ImportLog
        log = ImportLog(
            ontology_id=ontology_id,
            status=ImportLogStatus.RUNNING,
            cleaning_config=cleaning_config
        )
        db.add(log)
        await db.commit()

        # 2. 启动后台任务
        asyncio.create_task(
            self._run_import_task(
                ontology_id,
                log.id,
                cleaning_config
            )
        )

        return log.id
```

**写入 Neo4j**:
```python
async def _write_to_graph_db(
    self,
    ontology: Ontology,
    df: pd.DataFrame
):
    driver = get_neo4j_driver()

    async with driver.session() as session:
        for idx, row in df.iterrows():
            # 创建节点
            for cls in ontology.classes:
                node_props = {
                    "_ontology_id": ontology.id,
                    "_id": f"{ontology.id}_{cls.name}_{idx}"
                }

                # 映射属性
                for prop in cls.properties:
                    if prop.name in row.index:
                        node_props[prop.name] = row[prop.name]

                # 创建节点
                query = f"CREATE (n:{cls.name} $props)"
                await session.run(query, props=node_props)
```

**写入 Milvus**:
```python
async def _index_in_milvus(
    self,
    ontology_id: str,
    entity_id: str,
    entity_type: str,
    properties: dict
):
    # 生成内容文本
    content = " ".join([
        f"{k}: {v}"
        for k, v in properties.items()
    ])

    # 生成向量
    vector = await embed_text(content)

    # 插入 Milvus
    milvus = get_milvus_client()
    milvus.insert(
        collection_name=COLLECTION_NAME,
        data=[{
            "id": entity_id,
            "ontology_id": ontology_id,
            "entity_type": entity_type,
            "content": content,
            "vector": vector
        }]
    )
```

### 4.1 数据库表结构

```sql
CREATE TABLE data_sources (
    id VARCHAR(64) PRIMARY KEY,
    name VARCHAR(255) NOT NULL,
```

```sql
CREATE TABLE ontologies (
    id VARCHAR(64) PRIMARY KEY,
    name VARCHAR(255) NOT NULL,

    batch_id VARCHAR(64),        -- 批次ID
    classes JSONB NOT NULL,      -- 类定义数组
    relations JSONB,             -- 关系定义数组
    instances_count INTEGER,     -- 实例数量
    created_at TIMESTAMPTZ,
    updated_at TIMESTAMPTZ
);
```

## 7. 集成点说明

### 7.1 与现有本体管理器的集成

**导入完成后的跳转**:

```typescript
// frontend/src/pages/AnalysisReview.tsx

const handleImport = async (cleaningConfig) => {
  // ... 执行导入 ...

  // 导入完成后，跳转到导入日志或本体管理器

  // 方案 1: 跳转到导入日志
  navigate('/import-logs');

  // 方案 2: 跳转到本体管理器 (推荐)
  // navigate('/ontology-list');
};
```

**导入日志中查看本体**:

```typescript

  // 跳转到本体详情页面
  navigate(`/ontology/${log.ontology_id}`);

```

**建议的用户流程**:
```
数据接入 → 批量分析 → 分析审核 → 数据清洗 → 导入
                                              ↓
                                         导入日志
                                              ↓
                              ┌──────────────┴──────────────┐
                              ↓                             ↓
                         查看失败记录                   进入本体管理器
                         编辑并重试                     (现有系统)
```

### 7.2 与智能问答的集成

导入到本体管理器后，本体数据已写入 Neo4j 和 Milvus，可直接被智能问答系统使用。

**数据流**:
```
自动化本体发现
    ↓
导入 Neo4j + Milvus
    ↓
本体管理器 (查询、编辑)
    ↓
智能问答 (Cypher 查询 + 向量检索)
```

**无需额外集成工作**，因为：
- 本体结构保存在 PostgreSQL
- 图数据保存在 Neo4j
- 向量索引保存在 Milvus
- 智能问答系统直接使用这些数据

### 7.3 数据隔离与权限

**建议实现**:

1. **本体命名空间**
   - 自动化发现的本体标记 `source: 'auto_discovery'`
   - 人工创建的本体标记 `source: 'manual'`

2. **用户权限控制**
   - 自动化发现模块: 创建、编辑、删除自己的本体
   - 本体管理器: 管理所有本体（包括自动发现的）

3. **Neo4j 标签隔离**
   - 所有节点添加 `_ontology_id` 属性
   - 查询时按 `_ontology_id` 过滤

## 11. FAQ & 问题排查

### Q1: 分析任务一直卡在"运行中"状态？

**排查步骤**:
1. 检查后端日志: `tail -f backend.log`
2. 检查 DeepSeek API 是否正常
3. 检查数据源文件是否损坏
4. 尝试重新分析

### Q2: 导入失败，所有记录都失败？

**排查步骤**:
1. 检查 Neo4j 连接: `neo4j status`
2. 检查 Milvus 连接: `curl http://localhost:19530/healthz`
3. 查看导入日志的 `error_summary`
4. 检查本体定义是否正确（类名、属性名）

### Q3: 前端无法连接后端？

**排查步骤**:
1. 检查后端是否启动: `curl http://localhost:8000/health`
2. 检查前端代理配置: `vite.config.ts`
3. 检查 CORS 配置: `backend/app/main.py`
4. 查看浏览器控制台错误

### Q4: 数据清洗不生效？

**排查步骤**:
1. 检查清洗配置是否正确传递
2. 查看导入日志中的 `cleaning_config`
3. 检查 `DataCleaningEngine` 日志
4. 验证清洗算子名称是否正确
