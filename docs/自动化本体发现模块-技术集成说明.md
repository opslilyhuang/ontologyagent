# 自动化本体发现模块 — 技术集成说明

本文档面向技术团队，说明如何将「自动化数据接入 → 本体自动发现 → 人工确认 → 数据清洗 → 导入」这一套流程作为**独立一级菜单模块**接入现有产品，并与现有**本体管理**、**智能问答**融合。

---

## 一、背景与目标

### 1.1 现有产品能力（保持不变）

- **人工数据导入**：用户手动上传/配置数据源并完成编排
- **数据编排**：人工或半自动产出本体
- **本体管理**：本体的增删改查、发布、图结构、实体查看等
- **智能问答**：基于本体的对话式问答（单本体 Chat、多本体 Q&A 应用）

### 1.2 新增模块定位

将**自动化流水线**作为**一级菜单独立模块**接入：

- **入口**：一级菜单项（如「智能数据接入」或「自动化本体发现」）
- **流程**：数据接入 → 批量自动编排（本体发现）→ 人工审核/修改 → 可选数据清洗算子 → 点击「导入」→ **进入现有本体管理器**
- **融合点**：导入后的本体与现有「本体管理」「智能问答」共用同一套数据与能力，无两套体系。

---

## 二、一级菜单与模块划分

### 2.1 建议菜单结构

| 一级菜单       | 说明 |
|----------------|------|
| 仪表板         | 现有首页/概览 |
| **智能数据接入** | **新增**：自动化流水线入口（数据接入 + 分析审核 + 导入） |
| 本体管理       | 现有：本体列表、详情、图、单本体对话等 |
| 导入日志       | 现有或新增：按需展示导入任务与失败记录 |
| 智能问答       | 现有：多本体 Q&A 应用 |

「智能数据接入」即本方案设计的自动化模块，内部子流程见下节。

### 2.2 自动化模块内部流程（本方案设计）

```
[数据接入] → [批量分析/编排] → [分析审核] → [可选：数据清洗] → [导入] → [本体管理]
```

- **数据接入**：文件上传 / 数据库配置 / API 配置，创建数据源。
- **批量分析**：对选中数据源执行「自动编排」，产出**待审核本体**（draft）。
- **分析审核**：展示编排结果（类、属性、关系），支持人工修改、保存。
- **数据清洗（可选）**：导入前弹窗选择清洗算子（去空、去重、trim、校验等）。
- **导入**：按当前本体定义 + 清洗配置，将数据写入图库/向量库，**导入完成后本体进入现有本体管理**，可被智能问答等使用。

## 三、自动化流水线详细设计（本方案负责部分）

### 3.1 流程概览

```
┌─────────────────────────────────────────────────────────────────────────────┐
│  1. 数据接入 (Data Ingestion)                                                │
│     文件上传 / 数据库 / API → 创建数据源，列表展示                             │
└─────────────────────────────────────────────────────────────────────────────┘
                                        │
                                        ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│  2. 批量分析 (Batch Analyze)                                                  │
│     选择多个数据源 → 提交批量分析 → 得到 batch_id，轮询 batch 状态             │
└─────────────────────────────────────────────────────────────────────────────┘
                                        │
                                        ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│  3. 分析审核 (Analysis Review)                                                │
│     按 batch_id 展示本批次产出的本体列表；支持编辑类/属性/关系、保存            │
└─────────────────────────────────────────────────────────────────────────────┘
                                        │
                                        ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│  4. 数据清洗（可选）                                                          │
│     点击「导入」→ 弹窗选择清洗算子（可多选）→ 确认                              │
└─────────────────────────────────────────────────────────────────────────────┘
                                        │
                                        ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│  5. 导入 (Import)                                                             │
│     按本体 + 清洗配置执行导入 → 写 Neo4j/Milvus 等 → 跳转「导入日志」或「本体管理」│
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3.2 后端 API 清单（自动化模块相关）

统一前缀：`/api/v1`（或贵方网关前缀）。以下为**本方案已实现/约定的接口**，便于与现有后端对齐或对接。

#### 3.2.1 数据源 (data-sources)

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/data-sources/` | 创建数据源（type + config） |
| POST | `/data-sources/upload` | 单文件上传创建数据源 |
| POST | `/data-sources/upload/batch` | 多文件上传，每个文件一个数据源 |
| GET  | `/data-sources/` | 数据源列表 |
| GET  | `/data-sources/{source_id}` | 数据源详情 |
| GET  | `/data-sources/{source_id}/fields` | 数据源字段列表（用于审核页「从源表加属性」） |
| POST | `/data-sources/{source_id}/test-connection` | 测试连接（数据库/API） |
| POST | `/data-sources/{source_id}/analyze` | 单数据源分析（可选，与批量二选一） |
| POST | `/data-sources/batch-analyze` | **批量分析**，body: `{ "source_ids": ["id1","id2"] }`，返回 `{ batch_id, status, source_count }` |
| GET  | `/data-sources/batch/{batch_id}` | 查询批次状态与结果，返回 `batch_id, status, progress, current_stage, source_ids, ontology_ids, error` |
| GET  | `/data-sources/{source_id}/ontologies` | 某数据源已产出的本体列表（用于「查看本体」等） |
| DELETE | `/data-sources/{source_id}` | 删除数据源 |
| POST | `/data-sources/batch-delete` | 批量删除，body: `{ "source_ids": [] }` |

#### 3.2.2 本体 (ontologies) — 与现有本体管理共用

| 方法 | 路径 | 说明 |
|------|------|------|
| GET  | `/ontologies/` | 本体列表，可选 query `batch_id` 筛本批次 |
| GET  | `/ontologies/{ontology_id}` | 本体详情（类、关系、数据源等） |
| PUT  | `/ontologies/{ontology_id}` | 更新本体（审核页保存时用，主要更新 classes、relations） |
| POST | `/ontologies/{ontology_id}/generate-label` | 根据属性名生成中文 label/description（审核页用） |
| POST | `/ontologies/{ontology_id}/reanalyze` | 自然语言重新分析（可选） |
| **POST** | **`/ontologies/{ontology_id}/import`** | **执行导入**，body: `{ "cleaning_config": { "operators": [ { "name": "remove_nulls" }, ... ] } }`，返回 `{ import_log_id, status }` |

说明：导入接口会创建导入任务并异步执行，前端可跳转到「导入日志」页根据 `import_log_id` 查看进度与失败记录。

#### 3.2.3 导入日志 (import-logs)

| 方法 | 路径 | 说明 |
|------|------|------|
| GET  | `/import-logs/` | 导入日志列表 |
| GET  | `/import-logs/{log_id}` | 单条日志详情（状态、成功/失败数、错误摘要等） |
| GET  | `/import-logs/{log_id}/failed-records` | 失败记录分页，query: `page`, `page_size` |
| POST | `/import-logs/{log_id}/retry` | 用指定清洗配置重试失败记录 |
| DELETE | `/import-logs/{log_id}` | 删除日志（按需） |

#### 3.2.4 智能问答 (qa) — 与现有融合

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/qa/sessions` | 创建会话（选 ontology_ids） |
| GET  | `/qa/sessions` | 会话列表 |
| GET  | `/qa/sessions/{session_id}` | 会话详情 |
| PUT  | `/qa/sessions/{session_id}` | 更新会话（含 ontology_ids） |
| POST | `/qa/sessions/{session_id}/chat` | 发送消息 |

导入后的本体 ID 可直接用于上述 `ontology_ids`，与现有智能问答一致。

### 3.3 关键数据模型（本方案约定）

#### 数据源 DataSource

```ts
{
  id: string
  name: string
  type: 'file' | 'database' | 'api'
  status: 'created' | 'analyzing' | 'analyzed' | 'error'
  schema_info?: object
  error_message?: string
  created_at: string
  updated_at: string
}
```

#### 分析批次 AnalysisBatch

```ts
{
  batch_id: string
  status: 'pending' | 'running' | 'completed' | 'failed'
  progress: number
  current_stage: string
  source_ids: string[]
  ontology_ids: string[]
  error?: string
}
```

#### 本体 Ontology（与现有一致）

- `id, name, description, status, data_source_id, batch_id, classes, relations, instances_count, created_at, updated_at`
- `classes[]`: 每类含 `name, label, description, properties[]`
- `properties[]`: `name, type, label, display_name, is_primary_key, source` 等
- `relations[]`: `source_class, target_class, relation_name, cardinality` 等

#### 数据清洗配置 DataCleaningConfig

```ts
{
  operators: Array<{ name: string; params?: Record<string, any> }>
}
```

算子名称与后端一致，见下节。

#### 导入请求（本体导入接口 body）

```ts
{
  cleaning_config: {
    operators: [
      { name: "remove_nulls" },
      { name: "remove_duplicates" },
      { name: "trim_whitespace" },
      { name: "validate_email" }
      // 可选 params，如 { name: "fill_missing", params: { method: "constant", value: "" } }
    ]
  }
}
```

### 3.4 数据清洗算子（本方案已实现）

后端引擎按 `cleaning_config.operators` 顺序执行，每个算子可有可选 `params`。前端弹窗只需传算子 `name`（及部分算子 params），与下表对应即可。

| 算子 name | 说明 | 可选 params |
|-----------|------|-------------|
| `remove_nulls` | 去除空值行 | `columns`: 指定列，不传则全表 |
| `remove_duplicates` | 去除重复 | `columns`, `keep`: "first"/"last" |
| `fill_missing` | 填充缺失值 | `method`: "constant"/"forward"/"backward", `value`, `columns` |
| `trim_whitespace` | 文本前后空格 | `columns` |
| `normalize_case` | 统一大小写 | `columns`, `case`: "upper"/"lower" |
| `standardize_date` | 日期格式标准化 | `columns`, `format` |
| `validate_email` | 邮箱格式校验 | `columns`，校验失败进入 failed_records |
| `validate_phone` | 手机号格式校验 | `columns` |
| `detect_outliers` | 异常值检测 | `columns`, `method`: "iqr"/"zscore" |
| `mask_sensitive` | 数据脱敏 | `columns`, `mask_type`: "partial"/"full"/"hash" |

前端「数据清洗设置」弹窗可分组展示（基础清洗、数据规范化、数据验证、高级处理），用户勾选算子后组装为 `DataCleaningConfig` 在点击「导入」时传给 `POST /ontologies/{id}/import`。

### 3.5 前端路由与页面（本方案实现参考）

在**一级菜单「智能数据接入」**下，可对应以下路由与页面：

| 路由 | 页面 | 说明 |
|------|------|------|
| `/ingest` | 数据接入 | 文件/数据库/API 配置，数据源列表，勾选后「批量分析」→ 跳转 `/ingest/review/:batchId` |
| `/ingest/review/:batchId` | 分析审核 | 轮询 batch 状态；完成后展示本批次本体，可编辑类/属性/关系并保存；顶部「导入」打开数据清洗弹窗，确认后调用各本体 `import`，再跳转导入日志或本体管理 |
| `/ontologies` | 本体管理（现有） | 列表含「本批次导入」的本体，点击进详情/图/Chat |
| `/ontologies/:id` | 本体详情（现有） | 与现有一致 |
| `/ontologies/:id/chat` | 单本体对话（现有） | 与现有一致 |
| `/import-logs` | 导入日志 | 列表 + 详情 + 失败记录/重试（与现有或新增一致） |
| `/qa` | 智能问答（现有） | 多本体会话，可选入参包含本次导入的本体 |

**融合要点**：

- 审核页「保存」只调 `PUT /ontologies/{id}`，不改变现有本体模型。
- 审核页「导入」调 `POST /ontologies/{id}/import`，导入完成后该本体与现有「本体管理」中本体一致，可被 `/ontologies`、`/ontologies/:id/chat`、`/qa` 直接使用。
- 从「智能数据接入」进入的只是入口和流程不同，**落地后的本体数据与现有本体管理、智能问答共用同一套存储与 API**。

## 四、与现有系统的融合方式

### 4.1 本体管理

- **数据一致**：自动化模块产出的本体与人工创建/编辑的本体使用同一张表、同一套 API（`/ontologies/*`）。
- **状态**：编排产出为 draft；审核后保存仍为 draft；导入后可按业务需要自动/手动改为 published，与现有策略一致。
- **入口**：在「本体管理」列表中，可通过 `batch_id` 或 `data_source_id` 筛「来自智能数据接入」的本体，便于运营区分来源。

### 4.2 智能问答

- **数据一致**：导入完成后，新本体会出现在 `GET /ontologies/` 中，可直接加入 Q&A 会话的 `ontology_ids`。
- **无需改接口**：`/qa/sessions` 的创建、更新、对话与现有一致，仅需在选本体时把新导入的本体纳入可选列表。

### 4.3 权限与菜单

- 在现有权限体系下新增「智能数据接入」菜单与对应路由即可；是否与「数据源/本体/导入日志」做细粒度权限拆分由现有产品决定。

## 五、集成时技术注意点

1. **批量分析异步**：`batch-analyze` 返回后，前端需轮询 `GET /data-sources/batch/{batch_id}` 直至 `status === 'completed'` 或 `'failed'`，再进入审核页展示 `ontology_ids` 并拉取各本体详情。
2. **导入异步**：`POST /ontologies/{id}/import` 返回 `import_log_id` 后，导入在后台执行；前端可跳转到「导入日志」页，用 `import_log_id` 查状态与失败记录。
3. **审核页保存与导入顺序**：建议「导入」前先自动或提示用户执行一次保存（把所有编辑中的 classes/relations 写回 `PUT /ontologies/{id}`），再打开数据清洗弹窗并调用 import。
4. **BaseURL**：前端请求统一走 `/api/v1`（或贵方网关前缀），由网关或 Nginx 转发到本方案后端即可；若现有后端已提供部分接口，可做一层适配或合并路由。

## 六、小结

- **一级菜单**：新增「智能数据接入」，底下包含数据接入、分析审核（及可选的数据清洗）、导入，导入完成后进入**现有本体管理**与**智能问答**。
- **本方案负责**：数据接入与数据源 API、批量分析/批次状态、分析审核页的编辑与保存、数据清洗配置与算子、本体导入与导入日志 API；本体模型、本体管理、智能问答与现有产品共用，仅在使用方式上通过「智能数据接入」这条链路产出一批新本体并写入同一套存储。

按上述接口与数据模型对接，即可在不影响现有本体管理、智能问答的前提下，完成自动化模块的接入与融合。若现有后端路径或字段有差异，可在此基础上做一层薄适配（如路径别名、字段映射）即可。
